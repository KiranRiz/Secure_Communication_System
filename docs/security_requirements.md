# Security Requirements
## B9IS129 - Secure Communication System
### Written by Mubashir (security module owner)

---

When we were first reading through the assignment brief, the line that stood out most was this one: the server "MUST NOT be trusted not to leak any confidential information." That's a strange way to phrase it but what it means is - assume the server operator is watching everything. Design the system so that even if they are, it doesn't matter.

That one requirement drove most of our security decisions. Here's how we thought through it.

---

## The core problem we had to solve

The basic challenge is this: Alice and Bob want to talk privately. They have never met. There is a server in the middle that neither of them fully trusts. How do you set up encryption in that situation?

The answer we went with is ECDH - Elliptic Curve Diffie-Hellman key exchange. The way it works is that Alice and Bob each generate their own key pair (a private key they never share, and a public key they can share freely). They upload their public keys to the server. Then each of them downloads the other's public key and uses it - combined with their own private key - to independently calculate the same shared secret. The maths behind this means the shared secret cannot be worked out by anyone who only has the public keys, which is all the server ever sees.

We chose the X25519 curve specifically because it's the modern standard for this kind of key exchange (it's what Signal and WhatsApp use under the hood), it's fast, and it avoids some of the implementation traps that older curves had.

---

## Encryption: why AES-256-GCM

Once both sides have the shared secret, we derive an AES-256 key from it using HKDF (a standard key derivation function). Messages are then encrypted with AES-256-GCM.

We picked GCM mode over something like CBC for a specific reason: GCM gives you authenticated encryption. That means it doesn't just hide the content of a message - it also detects if anyone has tampered with it in transit. With CBC mode alone, an attacker could flip bits in the ciphertext and the decryption would still "succeed" but produce garbage
or manipulated data without the recipient knowing. GCM prevents that entirely - if a single byte has been changed, decryption fails loudly.

Every message uses a fresh randomly generated nonce (96 bits). This is important because reusing a nonce with the same key in GCM mode breaks the security completely - it's one of those things that's easy to get wrong in implementations, so we generate a new one every time.

---

## Identity verification: the fingerprint approach

The assignment lists several identity options. We went with what it calls "idiosyncratic identity verification" rather than PKI or social media based identity.

After key exchange, both users are shown a short fingerprint - a hash of their session key formatted into something human-readable like `a3f9:b821:9c4d:11e2`. The idea is that the two users can verify this fingerprint through any channel they already trust - a phone call, in person, a WhatsApp message. If both fingerprints match, they know no one
has inserted themselves in the middle of the key exchange.

The reason we preferred this over something like email-based verification is that it doesn't require trusting any third party at all. Email verification depends on trusting the email provider. Our fingerprint
approach only depends on the two users themselves. The trade-off is that it requires a manual step - the users have to actually check. We've noted that as a limitation in the README.

---

## Replay attack protection

A replay attack is when an attacker captures a legitimate encrypted message and resends it later to try to trick the server or recipient into processing it again. Because the message was genuinely encrypted, it would pass any encryption check.

Our fix for this is simple but effective: every message carries a UUID (a unique random identifier generated by the sender). The server keeps a record of message IDs it has seen in the last five minutes. If the same ID arrives again within that window, the message is dropped and the sender gets an error back. After five minutes the ID expires from memory, which stops the memory from growing forever on a long-running server.

---

## Password storage

User passwords are never stored in plain text, and we don't just hash them directly either. The reason for not doing a plain hash is rainbow tables - precomputed lists of hashes for common passwords that an attacker could use to reverse a stolen hash file almost instantly.

We add a random salt (16 bytes, generated fresh for each user at registration) to the password before hashing with SHA-256. Each user's salt is stored alongside their hash. This means even if two users pick the same password, their stored hashes will be completely different, and rainbow tables become useless.

---

## What the server can and cannot see

To be explicit about this, since it's central to the assignment requirements:

The server CAN see: usernames, public keys (these are designed to be public), the fact that user A sent a message to user B, and the approximate length of messages.

The server CANNOT see: message content (it only ever receives ciphertext), the shared secret (derived client-side, never transmitted), private keys (generated client-side in the browser, never leave the device).

This is what we mean when we say the server is a relay - it moves encrypted packets from one user to another without being able to read them.

## Algorithms summary

| Component | Algorithm | Why |
|---|---|---|
| Key exchange | ECDH X25519 | Modern standard, fast, secure |
| Key derivation | HKDF-SHA256 | Stretch shared secret into proper AES key |
| Message encryption | AES-256-GCM | Confidentiality + integrity in one step |
| Password hashing | SHA-256 + salt | Prevents rainbow table attacks |
| Replay protection | UUID + 5 min window | Blocks replayed captured messages |
| Fingerprint | SHA-256 truncated | Human-verifiable key identity |

All algorithms above are publicly documented standards - the system does not rely on any secret algorithm or security through obscurity, as required by the assignment brief.
